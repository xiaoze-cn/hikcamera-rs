错误模型有三个任务：

1. 把每个 SDK 返回码转成有类型的 Rust 错误（或 `Ok`）。
2. 表达 wrapper 自身的约束（找不到设备、ROI 非法、已在录像……），而不是
   编一个假的 SDK 码。
3. 出现新 SDK 码时容易扩展——往表里追加，不要写更多 match 分支。

## 三个组成

```mermaid
flowchart LR
    code["i32<br/>(SDK 返回值)"]
    status["Status(i32)<br/>newtype"]
    result["Result&lt;()&gt;<br/>(check)"]
    error["HikCameraError::Sdk<br/>{ status }"]
    info["StatusInfo<br/>(查表)"]

    code -->|Status::from| status
    status -->|check| result
    status -->|"err" or "ok"| error
    status -->|.info()| info
```

| 组成 | 位置 | 做什么 |
| --- | --- | --- |
| `Status(i32)` | `error.rs` | 原始 SDK 状态码 newtype。`Copy`、`From<i32>`、`Display` 打印成 `MV_E_NODATA (-2147483644, 0x80000007)`。 |
| `check(code: i32) -> Result<()>` | `error.rs` | 每一次 FFI 调用都经过的**唯一入口**。`MV_OK` 时返回 `Ok(())`，否则 `Err(HikCameraError::Sdk { status })`。 |
| `HikCameraError` 枚举 | `error.rs` | 每个 wrapper 层失败模式一个变体，外加承载 SDK 错误的 `Sdk { status }`。 |

## `HikCameraError`

```rust
#[derive(Debug, Clone, PartialEq, Eq, ThisError)]
#[non_exhaustive]
pub enum HikCameraError {
    Sdk { status: Status },
    NoDevice,
    DeviceNotFound { selector: String },
    MultipleDevices { selector: String, count: usize },
    NullHandle,
    SdkStatePoisoned,
    InvalidString { field: &'static str },
    UnsupportedNode { key: String, kind: &'static str },
    NodeValueMismatch { expected: &'static str, actual: &'static str },
    NodeInputMismatch { key: String, expected: &'static str, actual: &'static str },
    ValueOutOfRange { field: &'static str },
    RecordingInProgress,
    EmptyFrame,
    EmptyVideo,
    InvalidDuration { field: &'static str },
    InvalidFrameRate { field: &'static str },
    InvalidRoi,
}
```

两个关键不变量：

- `Sdk` 是**唯一**承载 SDK 状态码的变体。其它变体都是 wrapper 层约束
  （没匹配到设备、ROI 非法……），靠变体本身区分，不是靠魔法数字。
- `Display` 通过 `thiserror` 派生。无字段变体的 `#[error("...")]` 文案必须和
  `info().message` 一致——有一条单元测试
  （`display_matches_info_for_fieldless_variants`）守护这个。

## 为什么用 `Status` newtype？

SDK 码在 C 头文件里到处都是，但对 Rust 读者来说没有意义。newtype 给它一个
名字和一个稳定的边界：

```rust
pub struct Status(pub i32);

impl Status {
    pub const OK: Self = Self(sys::MV_OK as i32);
    pub const fn raw(self) -> i32 { self.0 }
    pub const fn as_u32(self) -> u32 { self.0 as u32 }
    pub const fn is_ok(self) -> bool { self.0 == Self::OK.0 }
    pub fn info(self) -> StatusInfo { /* 查表 */ }
}
```

边界之所以重要，是因为 bindgen 有个微妙问题：

<Aside type="caution" title="MV_* 常量是 u32，SDK 函数返回 int">
  C 头文件把状态码定义成 `0x80000000` 风格字面量。Bindgen 把它们生成
  `u32`（因为放不进 `i32`）。但 SDK 函数本身返回 `int`（带符号）。转换边界
  集中在 `Status::OK` 和 `check()`——别在别处撒 `as i32` / `as u32`。
</Aside>

## 常量查找表

把 SDK 状态码映射到人类可读的 `(name, message)` 是**数据**问题，不是逻辑
问题。所以以数据形式存放：

```rust
const SDK_ERRORS: &[(i32, StatusInfo)] = &[
    (sys::MV_OK as i32,              info("MV_OK", "Success")),
    (sys::MV_E_HANDLE as i32,        info("MV_E_HANDLE", "Invalid handle")),
    (sys::MV_E_SUPPORT as i32,       info("MV_E_SUPPORT", "Unsupported function")),
    // ... 还有约 250 条
];

fn sdk_error_info(status: Status) -> StatusInfo {
    SDK_ERRORS.iter()
        .find(|(code, _)| *code == status.0)
        .map(|(_, info)| *info)
        .unwrap_or(UNKNOWN_INFO)
}
```

新增 SDK 状态码时，**往 `SDK_ERRORS` 追加一行**。不要在别处加 match 分支。

## 访问器 API

```rust
let err = check(some_sdk_return)?;

// 拿到错误后：
let code: Option<i32>   = err.code();      // 仅 Sdk 变体返回 Some
let info: ErrorInfo     = err.info();      // (name, message) —— 所有变体都有
let name: &'static str  = err.name();      // 如 "MV_E_NODATA" 或 "DEVICE_NOT_FOUND"
let msg:  &'static str  = err.message();   // 如 "Timeout or no data received"
```

`code()` 故意返回 `Option<i32>`——wrapper 错误**没有** SDK 码，假装有就会抹掉
枚举本身已经在类型层面建立的区分。

## 何时新增 wrapper 变体

<Steps>

1. **确认不是 SDK 错误。** 如果 SDK 已经返回了状态码，`check()` 已经做了
   正确的事——不要新增变体。
2. **确认现有变体表达不了。** 新的 `InvalidRoi` 合理；新的
   `InvalidExposure` 不合理——曝光已经被 `NodeInputMismatch` 或
   `ValueOutOfRange` 覆盖。
3. **添加变体**，带命名字段（不要匿名 `String`）。
4. **加 `pub(crate)` 构造器**，和 `impl HikCameraError` 块里其它构造器并列。
5. **加 `info()` 分支**，给它稳定的 `name` 和 `message`。
6. **加 `#[error("...")]`**。如果变体无字段，跑
   `just test --lib error::tests::display_matches_info_for_fieldless_variants`——
   `Display` 文案和 `info().message` 漂移时这个测试会挂。

</Steps>

<Tabs>
  <TabItem label="推荐写法">
    - 命名字段承载**调用方真正需要的数据**（如 `selector`、`field`、
      `expected` / `actual`）。
    - 无字段变体表达"只有约束含义"的错误（`NullHandle`、`EmptyFrame`）。
    - 每个变体配一个 `pub(crate)` 构造器，调用方不用拼结构体字面量。
  </TabItem>
  <TabItem label="反模式">
    - 一个 `Other(String)` 兜底——废掉类型系统。
    - 给 wrapper 变体塞假的 SDK 码——污染码空间。
    - 为了 `info()` 给每个变体都加同一个字段——只在真用到的地方用命名字段。
  </TabItem>
</Tabs>
