`hikcamera-sys` binds the HikCamera MVS C SDK through one public header and one import library:

- Header entrypoint: `crates/hikcamera-sys/wrapper.h`
- C SDK header: `crates/hikcamera-sys/include/MvCameraControl.h`
- Import library: `crates/hikcamera-sys/lib/{win32,win64}/MvCameraControl.lib`
- Runtime DLL: `crates/hikcamera-sys/lib/{win32,win64}/MvCameraControl.dll`

<Aside type="note" title="Pruned vendor runtime">
  The repository no longer keeps every DLL shipped by the vendor SDK. It keeps the `MvCameraControl.dll` runtime path used by the current wrapper and removes GUI, DirectShow, GenTL producer, frame-grabber, serial-tool, and standalone media components.
</Aside>

## Build and runtime flow

<Steps>
1. `wrapper.h` includes `MvCameraControl.h`.
2. `MvCameraControl.h` pulls declarations from `CameraParams.h`, `PixelType.h`, `MvErrorDefine.h`, `MvISPErrorDefine.h`, and obsolete vendor headers.
3. `bindgen` generates Rust FFI declarations into Cargo's `OUT_DIR`.
4. `build.rs` links `MvCameraControl.lib` with `cargo:rustc-link-lib=dylib=MvCameraControl`.
5. The final Rust binary loads `MvCameraControl.dll` at runtime.
6. `MvCameraControl.dll` loads the retained GenICam, render, ISP, IPP, and vendor runtime DLLs.
</Steps>

## Header-to-runtime relationship

<CardGrid>
  <Card title="MvCameraControl.h">
    SDK entrypoints such as device enumeration, handle lifecycle, grabbing, node access, image save/conversion, recording, ISP helpers, and XML helpers. Runtime calls enter `MvCameraControl.dll` and its dependency chain.
  </Card>
  <Card title="CameraParams.h">
    Device, frame, node, image, save, recording, and transport structs/enums. These are compile-time layouts; runtime calls still go through `MvCameraControl.dll`.
  </Card>
  <Card title="PixelType.h">
    Pixel format constants used by frame and image APIs. Runtime handling lives in `MvCameraControl.dll` plus image/ISP support DLLs.
  </Card>
  <Card title="MvErrorDefine.h">
    `MV_E_*` SDK status constants. Rust exposes them through `Status` and the `StatusInfo` lookup table.
  </Card>
  <Card title="MvISPErrorDefine.h">
    `MV_ALG_*` ISP/algorithm status constants. ISP operations can require `MvISPControl.dll` and IPP DLLs.
  </Card>
  <Card title="Obsolete headers">
    `MvObsoleteInterfaces.h` and `ObsoleteCamParams.h` stay for bindgen compatibility, but they do not imply a separate runtime DLL.
  </Card>
</CardGrid>

## Removed runtime groups

| Group | Examples |
| --- | --- |
| GUI and wrapper helpers | `MvCameraControlGUI.dll`, `MvCameraControlWrapper.dll`, `MvCameraPatch.dll` |
| DirectShow filters | `MvDSS.ax`, `MvDSS2.ax` |
| GenTL producer plugins | `MvProducerGEV.cti`, `MvProducerU3V.cti`, `MvFGProducer*.cti` |
| Frame grabber control stack | `MVFGControl.dll`, `MvLCProducer.dll`, `MvCamLVision.dll` |
| Standalone transport/runtime helpers | `MvUsb3vTL.dll`, `MvSerial.dll`, `MvSerialCtrl.dll`, `MVGigEVisionSDK.dll` |
| Media/third-party codecs | `MediaProcess.dll`, `ThirdParty/*.dll`, `D3DCompiler_43.dll` |
| Unused C runtime variants | older VC90/VC120 DLLs not imported by the kept dependency chain |

<Aside type="caution" title="Future SDK surface can change this list">
  If a future wrapper API calls an SDK function that dynamically loads one of the removed components, restore that specific DLL and document the new dependency path here.
</Aside>
