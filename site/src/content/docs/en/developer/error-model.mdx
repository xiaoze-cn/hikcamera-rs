The error model has three jobs:

1. Turn every SDK return code into a typed Rust error (or `Ok`).
2. Express wrapper-level invariants (no device found, ROI invalid, recording
   in progress, …) without inventing fake SDK codes.
3. Stay easy to extend when a new SDK error code shows up — append to a
  table, don't write more match arms.

## The three pieces

```mermaid
flowchart LR
    code["i32<br/>(SDK return)"]
    status["Status(i32)<br/>newtype"]
    result["Result&lt;()&gt;<br/>(check)"]
    error["HikCameraError::Sdk<br/>{ status }"]
    info["StatusInfo<br/>(lookup)"]

    code -->|Status::from| status
    status -->|check| result
    status -->|"err" or "ok"| error
    status -->|.info()| info
```

| Piece | Where it lives | What it does |
| --- | --- | --- |
| `Status(i32)` | `error.rs` | Newtype for raw SDK status codes. `Copy`, `From<i32>`, `Display` prints `MV_E_NODATA (-2147483644, 0x80000007)`. |
| `check(code: i32) -> Result<()>` | `error.rs` | The **single entry point** every FFI call goes through. `Ok(())` on `MV_OK`, otherwise `Err(HikCameraError::Sdk { status })`. |
| `HikCameraError` enum | `error.rs` | One variant per wrapper-level failure mode, plus `Sdk { status }` for SDK errors. |

## `HikCameraError`

```rust
#[derive(Debug, Clone, PartialEq, Eq, ThisError)]
#[non_exhaustive]
pub enum HikCameraError {
    Sdk { status: Status },
    NoDevice,
    DeviceNotFound { selector: String },
    MultipleDevices { selector: String, count: usize },
    NullHandle,
    SdkStatePoisoned,
    InvalidString { field: &'static str },
    UnsupportedNode { key: String, kind: &'static str },
    NodeValueMismatch { expected: &'static str, actual: &'static str },
    NodeInputMismatch { key: String, expected: &'static str, actual: &'static str },
    ValueOutOfRange { field: &'static str },
    RecordingInProgress,
    EmptyFrame,
    EmptyVideo,
    InvalidDuration { field: &'static str },
    InvalidFrameRate { field: &'static str },
    InvalidRoi,
}
```

Two key invariants:

- `Sdk` is the **only** variant that carries an SDK status code. Every other
  variant is a wrapper-level invariant (no device matched, ROI invalid, …)
  and is discriminated by the variant itself, not a magic number.
- `Display` is derived via `thiserror`. The `#[error("...")]` text and
  `info().message` text must stay aligned for fieldless variants — there is
  a unit test (`display_matches_info_for_fieldless_variants`) that enforces
  this.

## Why a `Status` newtype?

Raw SDK codes are everywhere in the C headers but they mean nothing to a
Rust reader. The newtype gives them a name and a stable boundary:

```rust
pub struct Status(pub i32);

impl Status {
    pub const OK: Self = Self(sys::MV_OK as i32);
    pub const fn raw(self) -> i32 { self.0 }
    pub const fn as_u32(self) -> u32 { self.0 as u32 }
    pub const fn is_ok(self) -> bool { self.0 == Self::OK.0 }
    pub fn info(self) -> StatusInfo { /* table lookup */ }
}
```

The boundary matters because of a subtle bindgen wrinkle:

<Aside type="caution" title="MV_* constants are u32, SDK functions return int">
  The C headers define status codes as `0x80000000`-style literals. Bindgen
  emits them as `u32` because they don't fit in `i32`. But the SDK functions
  themselves return `int` (signed). The conversion boundary is consolidated
  in `Status::OK` and `check()` — don't sprinkle `as i32` / `as u32` casts
  elsewhere.
</Aside>

## The constant lookup table

Mapping SDK status codes to human-readable `(name, message)` pairs is a
**data** problem, not a logic problem. So it's stored as data:

```rust
const SDK_ERRORS: &[(i32, StatusInfo)] = &[
    (sys::MV_OK as i32,              info("MV_OK", "Success")),
    (sys::MV_E_HANDLE as i32,        info("MV_E_HANDLE", "Invalid handle")),
    (sys::MV_E_SUPPORT as i32,       info("MV_E_SUPPORT", "Unsupported function")),
    // ... ~250 more entries
];

fn sdk_error_info(status: Status) -> StatusInfo {
    SDK_ERRORS.iter()
        .find(|(code, _)| *code == status.0)
        .map(|(_, info)| *info)
        .unwrap_or(UNKNOWN_INFO)
}
```

To add a new SDK error code, **append one row** to `SDK_ERRORS`. Do not add
a new `match` arm anywhere.

## Accessor API

```rust
let err = check(some_sdk_return)?;

// On a known error variant:
let code: Option<i32>   = err.code();      // Some only for Sdk
let info: ErrorInfo     = err.info();      // (name, message) — every variant
let name: &'static str  = err.name();      // e.g. "MV_E_NODATA" or "DEVICE_NOT_FOUND"
let msg:  &'static str  = err.message();   // e.g. "Timeout or no data received"
```

`code()` returns `Option<i32>` deliberately — wrapper errors do **not** have
an SDK code, and pretending they do would erase the type-level distinction
the enum already encodes.

## When to add a new wrapper variant

<Steps>

1. **Confirm it's not an SDK error.** If the SDK already returned a status
   code, `check()` already did the right thing — don't add a new variant.
2. **Confirm it can't be expressed by an existing variant.** A new
   `InvalidRoi` is justified; a `InvalidExposure` is not — exposure is
   already covered by `NodeInputMismatch` or `ValueOutOfRange`.
3. **Add the variant** with named fields (no anonymous `String`).
4. **Add a `pub(crate)` constructor** alongside the other ones in the
   `impl HikCameraError` block.
5. **Add an `info()` arm** giving it a stable `name` and `message`.
6. **Add a `#[error("...")]`** attribute. Run
   `just test --lib error::tests::display_matches_info_for_fieldless_variants`
   if the variant is fieldless — the test will fail if your `Display` text
   drifts from `info().message`.

</Steps>

<Tabs>
  <TabItem label="Allowed patterns">
    - Named fields carrying **the data the caller needs** (e.g. `selector`,
      `field`, `expected` / `actual`).
    - Fieldless variants for invariant-only errors (`NullHandle`,
      `EmptyFrame`).
    - A `pub(crate)` constructor per variant so callers don't have to spell
      out the struct literal.
  </TabItem>
  <TabItem label="Anti-patterns">
    - A single `Other(String)` catch-all — defeats the type system.
    - Adding a fake SDK code to a wrapper variant — pollutes the code space.
    - Carrying the same field on every variant just to feed `info()` — use
      named fields only where they're actually consumed.
  </TabItem>
</Tabs>
